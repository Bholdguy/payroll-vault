1. overview

payrollvault is a single smart contract on the casper network that keeps a lifetime earnings total per agent id. it does one thing:

  • an authorized account records payments: record_payment(agent_id, amount)
  • anyone reads totals, free: get_earnings(agent_id)

it does not hold or move funds — it is a ledger, not a payment rail. no streaming, no escrow, no tokens. access control is enforced by the contract itself; any ui (including the site) is a read-only viewer.

live instance: contract package a9b847d4b2688ec1bdfd430f0a1a6488cb9395634aa680657a6ce6e5fd7579ee on casper testnet, deployed in this transaction.

2. contract reference (v1)

entry pointargsaccessbehavior
init() none runs once at deployment stores the deploying account as the sole authorized_caller
record_payment agent_id: String, amount: U256 authorized caller only adds amount to the agent's lifetime total with overflow-checked arithmetic, emits PaymentRecorded. reverts for any other caller before touching state.
get_earnings agent_id: String anyone, free returns the agent's lifetime total; returns 0 for unknown agents
authorized_caller none anyone, free returns the account allowed to record payments

amount is a raw unsigned integer (u256). the contract assigns it no currency — the operator decides what a unit means (cents, motes, credits) and should keep it consistent.

4. verify the live deployment

every claim is checkable without trusting this site:

  • the contract exists: open the contract package on testnet.cspr.live.
  • the deployment happened: open the deployment transaction — executed successfully, signed by the account that is now the authorized caller.
  • the state is real: with the repo checked out and livenet configured (section 6), reads are free: 
    cargo run --release --bin payroll_vault_cli -- contract PayrollVault get_earnings --agent_id "agent-001"
# INFO : Call result: 1000

5. build & test locally

prerequisites: rust nightly-2026-01-01 via rustup (pinned in rust-toolchain), the wasm target, cargo-odra, wasm-strip (wabt), wasm-opt (binaryen). linux or wsl recommended — parts of the casper toolchain do not compile on native windows.

# toolchain
rustup toolchain install nightly-2026-01-01
rustup target add wasm32-unknown-unknown --toolchain nightly-2026-01-01
cargo install cargo-odra --locked

# run the test suite (local vm, no network, no funds)
cargo odra test

# build the deployable wasm -> wasm/PayrollVault.wasm
cargo odra build

the v1 suite covers: the deployer becoming the authorized caller, payment accumulation across calls, the PaymentRecorded event, and the on-chain rejection of an unauthorized caller with state left untouched.

6. deploy your own instance

you need a casper account key (secret_key.pem) funded with testnet cspr from the faucet at testnet.cspr.live (deployment costs ~350 cspr in gas).

whichever key signs the deployment becomes the only account able to record payments on that instance — permanently, in v1. use a key you control and intend to keep.

export ODRA_CASPER_LIVENET_SECRET_KEY_PATH=/path/to/secret_key.pem
export ODRA_CASPER_LIVENET_NODE_ADDRESS=https://node.testnet.casper.network
export ODRA_CASPER_LIVENET_CHAIN_NAME=casper-test
export ODRA_CASPER_LIVENET_EVENTS_URL=https://node.testnet.casper.network/events
export ODRA_BACKEND=livenet

cargo run --release --bin payroll_vault_cli -- deploy

the cli prints the transaction hash and contract package hash, and saves the address to resources/contracts.toml so later commands find it. then:

# record a payment (write — costs gas, authorized key only)
cargo run --release --bin payroll_vault_cli -- contract PayrollVault record_payment \
  --agent_id "agent-001" --amount 1000 --gas "5 cspr"

# read it back (free)
cargo run --release --bin payroll_vault_cli -- contract PayrollVault get_earnings --agent_id "agent-001"

the cli offers to save your config to a .env file. keep that file out of version control — it contains the path to your secret key. the repo's .gitignore already excludes it.

7. v2: what's coming

v2 is written and tested (5/5 passing), not yet deployed. it keeps record_payment and get_earnings unchanged and adds:

  • owner role with two-step transfer — transfer_ownership(new_owner) proposes; the new owner must call accept_ownership(). a mistyped address can never permanently brick the contract.
  • operators — add_operator / remove_operator (owner only) grant teammates or bots the right to record without owning the contract; is_operator is a free read.
  • audit-trail events — PaymentRecorded gains a recorded_by field, and every permission change (OperatorAdded, OperatorRemoved, OwnershipTransferStarted, OwnershipTransferred) emits on-chain.

v2 source and tests live in this repo alongside v1.
